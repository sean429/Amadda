# Amadda 졸업논문 초안

## 요약

본 작품은 사용자가 작업을 중단한 뒤 다시 업무에 복귀할 때 필요한 핵심 작업 맥락을 저장하고 재확인할 수 있도록 돕는 로컬 기반 컨텍스트 복구 시스템 `Amadda`를 제안한다. 기존의 운영체제 또는 응용프로그램이 제공하는 복구 기능은 대체로 프로그램 실행 상태 일부만을 복원하거나 최근 문서 목록을 다시 제시하는 수준에 머무르는 경우가 많다. 그러나 실제 사용자는 단순히 프로그램을 다시 실행하는 것만으로는 이전 작업 흐름에 빠르게 재진입하기 어렵다. 사용자는 어떤 창을 열어 두었는지, 어떤 브라우저 탭을 보고 있었는지, 어떤 문맥에서 작업하고 있었는지를 함께 떠올려야 한다. 이러한 문제의식에서 출발하여, 본 작품은 작업 재개를 위한 최소하지만 실용적인 컨텍스트를 자동으로 수집하고 저장하는 데 초점을 맞추었다.

Amadda는 Windows 실행 프로세스 및 가시 창 제목을 수집하는 데스크톱 스냅샷 기능과, Chrome 확장을 통해 현재 브라우저 창의 URL 및 탭 제목을 수집하는 브라우저 스냅샷 기능으로 구성된다. 수집된 데이터는 FastAPI 기반 로컬 API를 통해 SQLite 데이터베이스에 저장되며, 사용자는 데스크톱 GUI에서 `save snapshot`, `restore latest snapshot`과 같은 명령을 입력하여 저장 및 일부 복구 기능을 실행할 수 있다. 또한 `Tracked Apps` 인터페이스를 제공하여 사용자가 추적 대상 앱을 직접 선택하도록 함으로써, 불필요한 백그라운드 프로세스를 제외하고 실제 작업과 관련된 항목 위주로 스냅샷을 구성하도록 설계하였다.

구현 결과, 본 시스템은 작업 중이던 주요 애플리케이션과 브라우저 탭 정보를 동일한 스냅샷 구조로 통합 저장할 수 있었으며, 최신 스냅샷에 포함된 URL을 다시 열어 브라우저 작업 흐름에 재진입하도록 지원할 수 있음을 확인하였다. 특히 단순한 애플리케이션 재실행이 아니라, 창 제목과 URL 같은 작업 단서를 함께 저장함으로써 사용자가 이전 작업을 더 빠르게 회상할 수 있는 기반을 마련하였다. 반면, 앱 내부 편집 상태의 완전한 복원이나 다중 브라우저 지원 등은 아직 구현되지 않았으며, 이는 향후 확장 과제로 남는다.

본 논문은 먼저 서론에서 연구 배경과 필요성, 목표 및 전체 구성을 설명한다. 이어서 관련연구에서는 컨텍스트 인식, 작업 재개, 활동 중심 컴퓨팅과 관련된 국내외 선행연구를 검토한다. 다음으로 제안 작품 소개에서는 Amadda의 이론적 배경과 시스템 구조를 설명하고, 구현 및 결과분석에서는 실제 구현 방식과 동작 결과를 분석한다. 마지막으로 결론 및 소감에서는 본 작품의 의의와 한계, 개발 과정에서 얻은 경험을 정리한다.

## 1. 서론

### 1.1 제안배경 및 필요성

현대의 컴퓨팅 환경에서 사용자는 하나의 작업만 수행하지 않는다. 문서 작성 중 브라우저로 자료를 검색하고, 메신저로 소통하며, 개발 도구로 코드를 수정하는 식으로 여러 응용프로그램과 창을 오가며 작업한다. 이 과정에서 사용자가 잠시 자리를 비우거나 시스템을 종료한 뒤 다시 작업을 이어가려 하면, 단순히 프로그램을 다시 실행하는 것만으로는 충분하지 않다. 실제로 필요한 것은 작업 당시의 문맥, 즉 어떤 앱을 열어 두었는지, 어떤 창이 핵심이었는지, 브라우저에서는 어떤 페이지를 참고하고 있었는지를 함께 확인하는 것이다.

기존 운영체제와 응용프로그램은 최근 문서 목록, 브라우저 세션 복구, 자동 저장 등의 기능을 제공하지만, 이러한 기능은 대체로 개별 애플리케이션 단위에 머문다. 사용자의 실제 작업은 여러 앱과 브라우저 탭에 걸쳐 분산되어 나타나기 때문에, 사용자는 작업 복귀 시 다시 기억을 더듬으며 환경을 재구성해야 한다. 이러한 반복은 작업의 몰입을 끊고, 복귀 시간을 증가시키며, 특히 자료 조사나 개발, 발표 준비처럼 다수의 창과 탭을 오가는 작업에서 비효율을 키운다.

컨텍스트 인식 시스템은 사용자의 상황, 위치, 행동, 사용 이력 등을 기반으로 적절한 정보나 서비스를 제공하는 방향으로 발전해 왔다[1][2][3]. 또한 인간-컴퓨터 상호작용 분야에서는 사용자가 중단된 작업으로 다시 진입하는 과정, 즉 task resumption을 지원하는 인터페이스와 기록 시스템의 중요성이 꾸준히 논의되어 왔다[4][5][6]. 이러한 흐름은 단순한 “앱 재실행”이 아니라 “작업 맥락 복구”가 실질적인 사용자 지원 방식이 되어야 함을 시사한다.

본 작품은 이러한 문제의식에 기반하여, 사용자의 작업 맥락을 로컬 환경에서 수집하고 저장한 뒤, 필요 시 해당 단서를 다시 제공함으로써 작업 재진입을 돕는 경량형 컨텍스트 복구 도구를 제안한다. 특히 졸업작품의 범위 안에서 구현 가능하면서도 실제 활용성이 있는 형태를 목표로 하여, Windows 실행 환경과 Chrome 브라우저를 중심으로 MVP(Minimum Viable Product)를 설계하였다.

### 1.2 연구논문/작품의 목표

본 연구의 1차 목표는 사용자의 작업 중단 시점에 존재하던 핵심 맥락 정보를 자동 또는 반자동으로 수집하여 저장하는 것이다. 여기서 핵심 맥락 정보란 실행 중인 주요 프로세스, 가시 창 제목, 브라우저 탭 제목 및 URL과 같이 사용자가 작업을 다시 떠올리는 데 직접적인 단서가 되는 정보를 의미한다. 이를 위해 사용자가 선택한 주요 앱만 추적하는 방식의 스냅샷 수집 기능과, 브라우저 탭 목록을 로컬 백엔드로 전송하는 확장 기능을 구현하였다.

2차 목표는 수집된 정보를 단일한 저장 구조로 관리하고, 이후 사용자가 가장 최근 스냅샷을 확인하거나 일부 복구할 수 있도록 하는 것이다. 본 작품에서는 프로세스, 창, 브라우저 탭을 모두 `snapshot_items`라는 공통 구조로 저장하도록 설계하였으며, 이를 통해 향후 YouTube 재생 시점, VS Code 작업 경로, 문서 파일 경로 등 더 정교한 컨텍스트로 확장 가능한 기반을 마련하였다.

3차 목표는 사용자가 직접 명령을 입력하고 결과를 확인할 수 있는 데스크톱 인터페이스를 제공하는 것이다. 이를 위해 GUI에서 자연어에 가까운 명령어를 규칙 기반으로 해석하고, 위험 명령은 권한 확인을 거친 뒤 실행하며, 일반 명령은 즉시 처리하는 구조를 도입하였다. 즉, Amadda는 단순한 데이터 수집기가 아니라 사용자의 작업 복귀를 지원하는 상호작용형 도구를 목표로 한다.

### 1.3 연구논문/작품 전체 overview

Amadda의 전체 구조는 크게 데스크톱 UI, 명령 해석기, 액션 디스패처, Windows 스냅샷 수집기, Chrome 확장, 로컬 API, SQLite 저장소의 7개 계층으로 구성된다. 데스크톱 UI는 사용자의 명령 입력과 로그 확인을 담당하며, 명령 해석기는 입력 문장을 `save snapshot`, `restore latest snapshot`, `open URL`, `sleep`, `shutdown` 등의 intent로 변환한다. 이후 디스패처는 해당 intent를 적절한 액션 서비스로 연결한다.

Windows 컨텍스트 수집은 실행 중인 프로세스와 보이는 창 제목을 대상으로 수행된다. 이때 모든 프로세스를 저장하는 대신, 사용자가 `Tracked Apps` 인터페이스에서 선택한 프로세스만 저장함으로써 잡음 데이터를 줄인다. 브라우저 컨텍스트 수집은 Chrome 확장에서 현재 창의 탭 목록을 읽어 로컬 FastAPI 서버의 `/browser/snapshot` 엔드포인트로 전송하는 방식으로 구현하였다. 저장소는 SQLite를 사용하며, 스냅샷 단위 메타데이터와 개별 항목 데이터를 분리하여 관리한다.

간단한 구현 결과를 요약하면, 본 작품은 로컬 환경에서 작업 중이던 앱과 창, 브라우저 탭 정보를 저장할 수 있고, 최신 스냅샷에 포함된 URL을 다시 열어 브라우저 작업 흐름을 일부 복원할 수 있었다. 특히 브라우저 탭과 Windows 프로세스를 동일한 데이터 구조 아래 저장함으로써, 향후 더 정교한 작업 복구 시스템으로 발전할 수 있는 구조적 기반을 확보하였다. 반면, 현재 복구 기능은 URL 재오픈 중심의 부분 복구에 해당하며, 완전한 세션 재현까지는 이르지 못한다.

이후 각 장의 구성은 다음과 같다. 2장에서는 본 작품과 관련된 컨텍스트 인식, 작업 재개, 활동 중심 컴퓨팅 관련 선행연구를 검토한다. 3장에서는 Amadda의 설계 원리와 시스템 구성을 설명한다. 4장에서는 실제 구현 내용과 실행 결과를 분석한다. 5장에서는 결론과 함께 프로젝트 수행 과정에서 얻은 경험과 향후 개선 방향을 정리한다.

## 2. 관련연구

컨텍스트 인식 컴퓨팅은 사용자의 현재 상황을 파악하여 적절한 정보나 서비스를 제공하는 연구 분야로, 모바일 환경과 지능형 사용자 지원 시스템의 발전과 함께 꾸준히 확장되어 왔다. 서효석과 이상용은 컨텍스트 인식 기반 개인화 서비스에서 사용자 행동 패턴을 추론하는 모델을 제안하며, 단순한 현재 상태 수집을 넘어 사용자의 목적과 행동 흐름을 파악하는 것이 중요하다고 보았다[1]. 이는 본 작품이 단순 프로세스 목록 저장이 아니라 “작업 재진입을 돕는 단서”를 저장하고자 한 방향과 연결된다. 또한 김성림과 권준희는 컨텍스트 인식 환경에서 정보 검색의 정확성과 신속성을 높이기 위한 레벨화된 정보 검색 기법을 제시하였으며, 사용자의 문맥에 따라 필요한 정보 접근 방식이 달라져야 함을 보여주었다[2]. 이는 작업 복귀 시에도 일괄적인 기록 제공이 아니라, 사용자가 실제로 필요로 하는 정보 중심의 맥락 제공이 중요하다는 점을 시사한다.

국내 연구 중 박재민과 이상용은 사용자의 컨텍스트를 기반으로 자동으로 문장을 생성하고 기록하는 시스템을 제안하면서, 사용자의 행동과 주변 문맥을 디지털 기록으로 축적하는 방식의 유용성을 보여주었다[3]. 본 작품은 자동 문장 생성까지는 포함하지 않지만, 추후 스냅샷 데이터를 기반으로 작업 흐름 요약이나 회고 문장을 생성하는 방향으로 확장할 수 있다는 점에서 유사한 문제의식을 공유한다.

국외 연구에서는 작업 전환과 재개를 지원하는 사용자 인터페이스 연구가 활발히 이루어졌다. Robertson 등은 `Task Gallery`를 통해 다수의 창과 문서를 작업 단위로 재배치하여 관리하는 3차원 윈도우 관리 방식을 제안하였고, 이는 사용자가 단순한 창 목록이 아니라 작업 중심 단위로 환경을 인식한다는 점을 보여주었다[4]. Dragunov 등은 `TaskTracer`를 제안하여 다중 작업을 수행하는 지식 노동자의 데스크톱 활동을 추적하고 작업별로 자원을 분류하는 환경을 구현하였다[5]. 이 연구는 사용자의 디지털 행위를 추적하여 작업 단위로 재구성할 수 있다는 점에서 본 작품의 개념적 기반이 된다.

웹 검색과 브라우징 문맥 복구에 관한 연구도 본 작품과 밀접하다. Morris 등은 `SearchBar`를 통해 검색 기록, 방문 페이지, 메모를 통합적으로 저장하여 사용자의 task resumption과 정보 재탐색을 지원하였으며, 다회차에 걸친 검색 과업에서 기존 브라우저 히스토리보다 더 나은 재획득 효과를 보였다고 보고하였다[6]. 이는 브라우저 탭의 URL과 제목을 구조적으로 저장하는 기능이 작업 재개에 실질적인 도움이 될 수 있음을 뒷받침한다. Bardram 등은 활동 중심 컴퓨팅(Activity-Based Computing)을 제안하면서, 전통적인 앱 중심 컴퓨팅은 사용자의 실제 활동 구조를 반영하지 못하며, 활동 단위를 기준으로 정보와 도구를 조직해야 한다고 주장하였다[7]. 본 작품 또한 파일이나 애플리케이션 자체보다 “작업 맥락”을 중심에 두고 있다는 점에서 이러한 관점과 맞닿아 있다.

선행연구를 종합하면, 사용자의 작업 흐름을 지원하기 위해서는 단순 저장보다 문맥화된 기록이 중요하며[1][2][3], 작업 단위의 재구성과 재개를 지원하는 인터페이스가 필요하고[4][5], 브라우저 및 검색 활동 역시 중요한 작업 맥락 요소임을 알 수 있다[6]. 또한 활동 중심 컴퓨팅 관점은 다양한 응용프로그램과 정보원을 하나의 작업 맥락으로 통합해 다루는 설계의 필요성을 제시한다[7]. 본 작품은 이러한 선행연구를 실제 데스크톱 환경에서 구현 가능한 로컬 도구의 형태로 축약한 사례라 할 수 있다.

다만 기존 연구들은 대체로 개념 모델, 대규모 사용자 인터페이스 실험, 혹은 특정 작업 추론 알고리즘에 초점을 두는 경우가 많다. 반면 본 작품은 졸업작품이라는 제약 안에서 실제 동작 가능한 경량 프로토타입을 구현하고, Windows 프로세스/창 정보와 브라우저 탭 정보를 단일 저장 구조로 통합했다는 점에서 차별성을 갖는다. 즉, 선행연구의 이론적 방향을 바탕으로, 제한된 범위 내에서 실용적인 작업 복귀 도구를 구현한 데 의의가 있다.

## 3. 제안 작품 소개

### 3.1 작품 개요

Amadda는 사용자가 현재 수행 중인 작업과 관련된 앱, 창, 브라우저 탭의 정보를 하나의 스냅샷으로 저장하고, 이후 작업을 다시 시작할 때 그 정보를 활용해 빠르게 이전 흐름에 재진입하도록 돕는 데스크톱 기반 시스템이다. 본 작품은 완전한 운영체제 상태 복제를 목표로 하지 않는다. 대신 사용자가 “무엇을 하고 있었는지”를 즉시 떠올리게 만드는 작업 단서를 수집하고 구조화하는 데 집중한다.

이 작품의 핵심 설계 철학은 다음 세 가지이다. 첫째, 로컬 중심 설계를 통해 개인정보가 외부 서버로 전송되지 않도록 한다. 둘째, 실제 사용성이 높은 최소 기능부터 구현하여 점진적으로 확장 가능한 구조를 만든다. 셋째, 앱 중심이 아닌 작업 맥락 중심의 저장 구조를 채택한다. 이러한 철학은 시스템 복잡도를 과도하게 높이지 않으면서도, 향후 브라우저 외의 문서 앱, 개발 도구, 멀티미디어 앱 등으로 확장 가능한 기반을 제공한다.

### 3.2 이론적 배경

본 작품의 이론적 배경은 크게 컨텍스트 인식과 활동 중심 컴퓨팅 두 축으로 설명할 수 있다. 컨텍스트 인식 관점에서 중요한 것은 사용자의 현재 환경을 단순 센서 데이터가 아니라 의미 있는 상황 정보로 해석하는 것이다[1][2]. 예를 들어 `chrome.exe`가 실행 중이라는 사실만으로는 사용자의 작업 목적을 알기 어렵지만, 특정 탭 제목과 URL, 그리고 함께 열려 있는 문서 편집기나 개발 도구의 창 제목을 결합하면 작업 맥락을 보다 구체적으로 파악할 수 있다. 따라서 본 작품은 프로세스명, 창 제목, URL, 활성 탭 여부와 같은 속성을 함께 저장한다.

활동 중심 컴퓨팅 관점에서는 사용자의 실제 관심 단위가 애플리케이션이 아니라 “활동” 또는 “작업”이라는 점이 중요하다[7]. 사용자는 발표 자료 작성, 보고서 조사, 프로그래밍 디버깅처럼 작업 단위로 사고하지만, 기존 컴퓨팅 환경은 앱과 파일 단위로 인터페이스를 제공한다. Amadda는 여러 앱과 탭을 하나의 스냅샷 안에 담아 둠으로써, 분산된 정보를 다시 하나의 작업 흐름으로 묶어 재인식할 수 있도록 한다.

### 3.3 시스템 구성

Amadda의 시스템은 데스크톱 클라이언트, 명령 처리 계층, 스냅샷 수집 계층, 로컬 백엔드, 데이터 저장 계층, 브라우저 확장 계층으로 구성된다.

첫째, 데스크톱 클라이언트는 PySide6 기반 GUI로 구현되었으며, 텍스트 명령 입력창과 로그 출력 영역, 그리고 `Tracked Apps` 선택 인터페이스를 제공한다. 사용자는 이 GUI에서 스냅샷 저장, 최신 스냅샷 복구, URL 열기, 절전 및 종료 명령 등을 실행할 수 있다.

둘째, 명령 처리 계층은 규칙 기반 intent parser와 dispatcher로 구성된다. parser는 `save snapshot`, `restore latest snapshot`, `open ...`, `sleep`, `shutdown` 등의 명령을 intent로 변환하며, 일부 한국어 표현도 인식한다. dispatcher는 변환된 intent를 스냅샷 저장, 브라우저 복구, 시스템 명령 실행과 같은 구체적 서비스에 연결한다.

셋째, 스냅샷 수집 계층은 Windows 환경에서 실행 중인 프로세스와 가시 창 제목을 수집한다. 이때 모든 프로세스를 무차별적으로 저장하지 않고, 시스템 프로세스와 백그라운드 프로세스를 제외하는 필터를 적용한다. 또한 사용자가 선택한 tracked app만 최종 저장 대상으로 인정함으로써, 실제 작업에 무관한 잡음을 줄인다.

넷째, 로컬 백엔드는 FastAPI로 구현되었다. 이 백엔드는 데스크톱 UI와 함께 실행되며, `POST /command`, `GET /snapshots/latest`, `POST /browser/snapshot`와 같은 엔드포인트를 제공한다. 특히 브라우저 확장이 이 로컬 API를 통해 탭 정보를 전송하도록 함으로써, 데스크톱 앱과 브라우저 확장이 느슨하게 결합되도록 설계하였다.

다섯째, 데이터 저장 계층은 SQLite를 사용한다. 스냅샷 메타데이터는 `snapshots` 테이블에 저장되고, 개별 항목은 `snapshot_items` 테이블에 저장된다. 각 항목은 `app_name`, `title`, `url`, `item_type`, `process_name`, `executable_path`, `created_at` 등의 속성을 가진다. 또한 사용자가 추적 대상으로 선택한 앱은 `tracked_processes` 테이블에 저장된다.

여섯째, 브라우저 확장 계층은 Chrome Manifest V3 기반으로 구현되었다. 확장 아이콘을 클릭하면 현재 창의 탭 목록을 수집하여 로컬 FastAPI 백엔드로 전송하고, 성공 여부를 배지 텍스트로 피드백한다. 이 구조는 향후 YouTube 재생 시점, 사이트별 메타데이터, 다중 브라우저 지원 등으로 확장하기에 적합하다.

### 3.4 데이터 흐름

Windows 스냅샷 저장 흐름은 다음과 같다. 사용자가 GUI에서 `save snapshot` 명령을 입력하면 parser가 이를 `save_snapshot` intent로 해석하고, dispatcher가 snapshot action service를 호출한다. 이후 Windows snapshot collector가 현재 실행 중인 프로세스 목록과 가시 창 제목을 수집하고, tracked app 조건 및 필터 규칙을 적용하여 `SnapshotItem` 목록을 구성한다. 이 데이터는 repository를 통해 SQLite에 저장된다.

브라우저 스냅샷 저장 흐름은 다음과 같다. 사용자가 Chrome 확장 아이콘을 클릭하면 service worker가 현재 창의 탭을 조회하여 `url`, `title`, `active` 값을 추출한다. 이 정보는 `POST /browser/snapshot` 요청으로 로컬 API에 전달되고, 백엔드는 각 탭을 `item_type="browser_tab"`인 스냅샷 항목으로 변환하여 저장한다.

최신 스냅샷 복구는 현재 부분적으로 구현되어 있다. 사용자가 `restore latest snapshot` 명령을 입력하면 최신 스냅샷을 조회한 뒤, 그 안에 포함된 URL들을 기본 웹브라우저로 다시 여는 방식으로 동작한다. 즉 현재 단계의 복구는 “브라우저 URL 재오픈” 중심이며, Windows 애플리케이션의 세부 상태나 창 배치를 완전하게 복원하는 수준은 아니다.

### 3.5 설계상의 특징

본 작품의 첫 번째 특징은 서로 다른 컨텍스트 소스를 동일한 저장 구조로 통합했다는 점이다. 프로세스, 창, 브라우저 탭은 성격이 다르지만, 모두 `snapshot_items`라는 공통 테이블에서 관리된다. 이를 통해 데이터 저장 구조는 단순해지고, 새로운 컨텍스트 소스가 추가되더라도 전체 시스템 구조를 크게 바꾸지 않아도 된다.

두 번째 특징은 사용자 선택 기반의 tracked app 메커니즘이다. 일반적인 스냅샷 시스템은 시스템 전체를 기록하려고 시도하지만, 이는 데이터 양 증가와 노이즈 문제를 유발한다. 반면 Amadda는 사용자가 현재 중요하다고 판단한 앱만 추적 대상으로 저장하므로, 보다 실용적인 컨텍스트만 남길 수 있다.

세 번째 특징은 로컬 중심 아키텍처이다. 브라우저 확장과 데스크톱 앱, 데이터베이스가 모두 로컬 환경에서 동작하므로 민감한 작업 기록이 외부 서버로 전송되지 않는다. 이는 사용성뿐 아니라 프라이버시 측면에서도 장점이 있다.

## 4. 구현 및 결과분석

### 4.1 구현 환경 및 주요 기술

본 작품의 구현에는 Python 기반 데스크톱 및 백엔드 기술과 JavaScript 기반 브라우저 확장 기술이 사용되었다. 데스크톱 인터페이스는 PySide6로 구현하였고, 로컬 API 서버는 FastAPI를 사용하였다. 데이터 저장은 SQLite로 구성하여 별도의 외부 DB 서버 없이 로컬에서 즉시 실행 가능하도록 하였다. Windows 프로세스 및 창 정보 수집에는 `psutil`과 Win32 API 기반 처리 방식을 사용하였으며, 브라우저 측은 Chrome Manifest V3 구조로 작성하였다.

이와 같은 기술 선택은 졸업작품 수준에서의 구현 용이성과 실제 시연 가능성을 모두 고려한 결과이다. Python은 GUI, 로컬 서버, 데이터 처리 로직을 하나의 생태계 안에서 통합하기에 적합했고, SQLite는 배포와 실행이 간단하며 구조 검증에 충분했다. Chrome 확장은 브라우저 탭의 URL과 제목을 운영체제 수준보다 더 정확하게 얻기 위해 사용되었다.

### 4.2 Windows 스냅샷 구현

Windows 스냅샷 수집기는 현재 실행 중인 프로세스를 열거하고, 각 프로세스에 연결된 가시 창 제목을 수집한 뒤, tracked app 조건 및 필터 규칙을 적용하여 최종 항목을 생성한다. 시스템 프로세스, 서비스성 프로세스, 보안 프로그램, 백그라운드 런처 등은 기본적으로 제외하며, 창이 보이는 일반 애플리케이션이나 `chrome.exe`, `Code.exe`, `explorer.exe` 등 대표적인 작업 앱은 우선 고려 대상으로 삼는다.

실제 저장 시에는 한 프로세스에 대해 `process` 항목 하나와, 존재하는 각 가시 창 제목에 대해 `window` 항목을 별도로 생성한다. 예를 들어 하나의 개발 도구가 여러 창을 가지고 있다면, 해당 프로세스는 하나의 프로세스 항목과 여러 개의 창 항목으로 기록된다. 이 방식은 동일한 응용프로그램의 존재 여부와 사용자가 실제로 보던 창 제목을 함께 저장할 수 있다는 장점이 있다.

또한 GUI의 `Tracked Apps` 대화상자는 현재 실행 중인 프로세스를 그룹화하여 보여주며, 사용자가 체크한 항목만 `tracked_processes` 테이블에 저장한다. 이후 실제 스냅샷 저장 시에는 이 추적 목록과 일치하는 프로세스만 포함되므로, 개인 PC에서 흔히 많이 보이는 백그라운드 잡음 데이터를 줄일 수 있다. 이는 단순하지만 실제 사용성 측면에서 중요한 기능으로 판단된다.

### 4.3 브라우저 스냅샷 구현

브라우저 스냅샷 기능은 Chrome 확장과 FastAPI 엔드포인트의 협력으로 구현되었다. 확장 프로그램은 아이콘 클릭 이벤트를 감지하면 `chrome.tabs.query({ currentWindow: true })`를 호출하여 현재 창의 탭 목록을 수집한다. 이후 각 탭에서 `url`, `title`, `active` 값을 추출하여 JSON으로 직렬화한 뒤 로컬 API에 전송한다.

백엔드는 이 요청을 수신하면 각 탭을 `SnapshotItem` 객체로 변환하여 데이터베이스에 저장한다. 이때 `item_type`은 `browser_tab`으로 설정하고, `title`에는 활성 탭 여부를 표시하며, `process_name`은 `chrome`으로 저장한다. 따라서 브라우저 데이터 또한 다른 스냅샷 항목과 동일한 구조 아래서 조회할 수 있다.

이 구현의 장점은 운영체제 레벨에서 파악하기 어려운 정확한 URL 정보를 브라우저 내부 API를 통해 얻을 수 있다는 점이다. 단지 `chrome.exe`가 실행 중이라는 사실만으로는 사용자가 어떤 웹페이지를 보고 있었는지 알 수 없지만, 확장을 통해 수집한 탭 정보는 사용자의 실제 작업 문맥을 직접적으로 보여준다.

### 4.4 저장 구조 및 복구 구현

스냅샷 저장 구조는 `snapshots`와 `snapshot_items`의 2계층 구조로 구성하였다. `snapshots`는 스냅샷 단위의 생성 시각과 식별자를 저장하고, `snapshot_items`는 실제 프로세스, 창, 브라우저 탭 등의 항목을 저장한다. 이 구조는 하나의 스냅샷 안에 서로 다른 종류의 컨텍스트를 동시에 포함할 수 있게 해 준다.

또한 저장소는 최신 스냅샷을 일정 개수만 유지하도록 하여 오래된 스냅샷을 자동 정리한다. 이를 통해 장기 사용 시 데이터 누적을 제어하고, 최신 작업 중심의 기록을 유지하도록 하였다. tracked app 정보는 별도 테이블에서 관리하며, 사용자의 선택 변경 시 해당 목록 전체를 갱신하는 방식으로 단순하게 구현하였다.

복구 기능은 현재 가장 최신 스냅샷을 조회한 뒤, URL이 존재하는 항목들을 기본 브라우저에서 다시 여는 수준으로 구현하였다. 따라서 현재 단계에서 “복구”란 완전한 세션 복원이 아니라, 사용자가 다시 참고해야 할 웹 문서를 빠르게 재오픈하도록 지원하는 것이다. 이는 기능 범위는 제한적이지만, 적어도 브라우저 기반 조사나 자료 탐색 작업의 재개를 지원하는 데 의미가 있다.

### 4.5 결과 및 분석

본 작품의 구현 결과, Windows 환경에서 실행 중인 앱과 가시 창 제목을 저장하고, Chrome 현재 창의 브라우저 탭 정보를 별도의 확장을 통해 저장할 수 있는 통합 구조를 구축하였다. 특히 데스크톱 GUI, 로컬 API, 브라우저 확장, SQLite 저장소가 하나의 로컬 시스템으로 결합되어 실제 동작 가능한 프로토타입을 형성했다는 점이 성과라 할 수 있다.

결과를 기능별로 분석하면 다음과 같다. 첫째, 작업 앱 선택 기반 스냅샷 저장은 모든 프로세스를 기록하는 방식보다 실제 활용성이 높았다. 사용자가 추적 대상을 스스로 고를 수 있기 때문에 저장 결과가 더 읽기 쉬워지고, 이후 복귀 시에도 핵심 앱 중심으로 작업 흐름을 회상하기 쉬워진다. 둘째, 브라우저 탭의 URL과 제목을 저장함으로써 단순 프로세스 목록만으로는 얻을 수 없는 구체적 문맥 단서를 제공할 수 있었다. 셋째, 서로 다른 유형의 데이터를 동일한 스냅샷 구조로 저장함으로써 향후 확장 가능성을 확보하였다.

반면 한계도 분명하다. 우선 현재 스냅샷 복구는 URL 재오픈 중심으로만 제공되기 때문에, 사용자가 작업하던 애플리케이션의 내부 상태나 파일 편집 지점까지 복원하지는 못한다. 또한 브라우저는 현재 Chrome 창만 수집하며, 자동 주기 저장이나 YouTube 시청 시점 저장도 아직 포함하지 않는다. Windows 측 필터 또한 휴리스틱 기반이므로 사용자 환경에 따라 불필요한 항목이 섞이거나 필요한 항목이 빠질 가능성이 있다.

그럼에도 불구하고 본 작품은 “완전한 복원”이 아닌 “실용적인 컨텍스트 복구”라는 목표에는 부합한다. 사용자가 다시 기억해야 하는 핵심 단서를 빠르게 수집하고 제공한다는 점에서, 실제 사용 시 작업 재개 시간을 줄이는 기반이 될 수 있다. 특히 발표 준비, 자료 조사, 개발 작업처럼 다수의 앱과 브라우저 탭을 함께 활용하는 상황에서 유용성이 높을 것으로 판단된다.

## 5. 결론 및 소감

본 논문에서는 작업 중단 이후 사용자의 빠른 업무 복귀를 지원하기 위한 로컬 기반 컨텍스트 복구 시스템 `Amadda`를 제안하고 구현하였다. 본 작품은 Windows 프로세스 및 창 제목, Chrome 탭의 URL과 제목을 수집하여 SQLite에 통합 저장하고, 사용자가 최근 작업 흐름을 다시 떠올릴 수 있도록 돕는 것을 목표로 하였다. 구현 결과, 데스크톱 UI, 로컬 API, 브라우저 확장, 데이터 저장소를 결합한 실제 동작 가능한 MVP를 완성하였으며, 브라우저 URL 재오픈을 통한 부분 복구 기능도 확인할 수 있었다.

본 작품의 의의는 완전한 세션 복제가 아닌, 사용자의 작업 재진입에 필요한 핵심 단서를 복구하는 실용적 접근을 제시했다는 데 있다. 기존의 앱 중심 복구 기능이 놓치기 쉬운 창 제목, 브라우저 탭, 사용자 선택 기반 주요 앱 정보 등을 함께 다루어 작업 중심 관점에 더 가까운 시스템을 구현하였다. 또한 다양한 컨텍스트를 공통 스키마로 저장하는 구조를 채택함으로써, 향후 YouTube 재생 시점, VS Code 작업 경로, 문서 파일 위치 등 더 풍부한 맥락 정보를 추가할 수 있는 기반을 마련하였다.

프로젝트를 진행하면서 가장 크게 느낀 점은, 사용자가 실제로 필요로 하는 복구 정보와 시스템이 기술적으로 수집하기 쉬운 정보가 반드시 같지 않다는 점이었다. 예를 들어 프로세스 목록을 수집하는 것은 비교적 단순했지만, 사용자가 정말 다시 보고 싶은 것은 “어떤 페이지를 보았는지”, “어떤 창에서 무슨 작업을 했는지”와 같은 더 구체적인 문맥이었다. 이 때문에 단순 구현보다 어떤 정보가 사용자에게 의미 있는지를 고민하는 과정이 중요했다.

또한 데스크톱 앱, 로컬 API, 브라우저 확장, 데이터베이스를 하나의 작품으로 통합하는 과정에서 각 구성 요소가 어떻게 분리되고 연결되어야 하는지에 대해 실질적인 설계 경험을 얻을 수 있었다. 특히 처음에는 기능 추가에만 집중하기 쉬웠지만, 작업이 진행될수록 저장 구조를 단순하게 유지하고 확장 가능성을 확보하는 것이 더 중요하다는 점을 체감하였다. 이 경험은 이후 실제 서비스나 응용프로그램을 설계할 때도 큰 도움이 될 것이라 생각한다.

아쉬운 점도 있다. 현재 구현은 복구 기능이 제한적이며, 작업 요약, 자동 저장, 앱별 세부 상태 복원 같은 고도화 기능까지는 도달하지 못했다. 그러나 이러한 한계는 동시에 향후 발전 방향이기도 하다. 추후에는 브라우저 자동 스냅샷, YouTube 재생 시점 저장, 문서 및 IDE 맥락 복구, LLM 기반 작업 요약 기능 등을 추가함으로써 보다 완성도 높은 작업 복귀 시스템으로 확장할 수 있을 것이다.

종합하면, 본 작품은 사용자의 “아, 맞다”를 대신 떠올려 주는 도구라는 출발점에서 시작하여, 실제로 동작하는 컨텍스트 복구 프로토타입을 구현했다는 점에서 의미가 있다. 졸업작품을 수행하며 단순한 아이디어 제시를 넘어 실제 구현과 구조 설계, 기능 한계 분석까지 경험할 수 있었고, 이를 통해 사용자 중심 소프트웨어 설계의 중요성을 깊이 이해하게 되었다.
