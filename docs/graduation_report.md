# Amadda: 로컬 기반 작업 컨텍스트 복구 시스템

## 요약

현대의 컴퓨팅 환경에서 사용자는 문서 작성, 웹 검색, 코드 편집 등 여러 앱과 브라우저 탭을 동시에 사용하며 작업한다. 작업을 중단한 뒤 다시 재개할 때, 단순히 프로그램을 다시 실행하는 것만으로는 이전 흐름을 즉시 복원하기 어렵다. 사용자는 어떤 창을 열어 두었는지, 어떤 탭을 참고하고 있었는지, 어떤 프로젝트를 작업하고 있었는지를 스스로 다시 떠올려야 한다.

본 작품 `Amadda`는 사용자의 작업 맥락을 로컬 환경에서 자동으로 수집, 저장하고, 이후 작업 재개를 돕는 데스크톱 기반 컨텍스트 복구 시스템이다. Amadda는 Windows 실행 프로세스·가시 창 제목 수집, Chrome 브라우저 탭 자동 수집, VS Code 워크스페이스 및 Word 문서 컨텍스트 보강, LLM 기반 작업 요약, 로컬 Whisper 음성 입력, 규칙 기반 명령어 시스템, 스냅샷 히스토리 UI, 작업 복구 기능을 하나의 통합된 로컬 도구로 제공한다.

수집된 컨텍스트는 SQLite에 저장되며, 사용자는 텍스트 또는 음성으로 명령을 입력해 스냅샷 저장, 과거 작업 요약, URL 및 VS Code 워크스페이스 재오픈, 앱 실행, 웹 검색 등을 실행할 수 있다. 모든 기능은 외부 서버 없이 로컬에서 동작하며, 개인정보가 외부로 전송되지 않도록 설계하였다.

## 1. 서론

### 1.1 제안 배경 및 필요성

현대 지식 노동자의 작업 환경은 단일 앱 안에 머물지 않는다. 발표 자료를 작성하면서 관련 자료를 브라우저로 검색하고, 코드를 작성하는 중에 Stack Overflow 탭을 열어 두며, 보고서를 쓰다가 메신저로 소통하는 식으로 여러 앱과 창을 넘나드는 것이 일반적인 작업 방식이다. 이 과정에서 잠깐 자리를 비우거나 컴퓨터를 종료한 뒤 다시 작업에 복귀하면, 작업 흐름을 재구성하는 데 적지 않은 시간이 소요된다.

기존 운영체제와 애플리케이션이 제공하는 복구 기능은 대부분 앱 단위에 국한된다. 브라우저는 세션 복구로 탭을 다시 열어 주지만, 어떤 탭이 핵심이었는지는 알 수 없다. VS Code는 마지막 열린 파일을 기억하지만, 그 파일이 어떤 작업의 일부였는지는 기록하지 않는다. 이러한 파편화된 복구 도구들은 사용자가 이전 작업으로 즉시 복귀하기 어렵게 만든다.

컨텍스트 인식 컴퓨팅 분야의 선행연구들은 사용자의 상황 정보를 활용해 더 지능적인 지원을 제공하는 방향으로 발전해 왔다[1][2][3]. 또한 인간-컴퓨터 상호작용 분야에서는 중단된 작업으로의 재진입(task resumption)을 돕는 도구와 인터페이스의 필요성이 꾸준히 논의되어 왔다[4][5][6]. 이러한 연구 흐름은 "앱 재실행"이 아닌 "작업 맥락 복구"가 실질적인 사용자 지원이 되어야 함을 공통적으로 시사한다.

본 작품은 이러한 문제의식에서 출발하여, 사용자의 작업 맥락을 자동으로 수집하고 저장한 뒤, 필요 시 해당 단서를 다시 제공함으로써 작업 재진입을 돕는 로컬 기반 도구를 제안한다.

### 1.2 작품 목표

본 작품의 목표는 다음 세 가지이다.

**첫째, 작업 맥락의 자동 수집이다.** 사용자가 별도의 행동을 취하지 않아도, 앱 실행 시부터 일정 주기(15분)마다 현재 작업 중인 Windows 프로세스, 가시 창 제목, 브라우저 탭 목록을 자동으로 스냅샷으로 저장한다.

**둘째, 수집된 컨텍스트를 활용한 작업 요약과 복구이다.** 사용자는 음성 또는 텍스트로 "뭐 하고 있었어?"라고 물으면 LLM이 최근 스냅샷을 분석하여 작업 내용을 요약해 준다. 또한 이전에 열어 두었던 브라우저 탭과 VS Code 워크스페이스를 한 번에 다시 열어 작업 흐름을 복원할 수 있다.

**셋째, 자연어 기반의 상호작용 도구 제공이다.** 텍스트 입력과 오프라인 음성 인식을 통해 앱 실행, 웹 검색, 사이트 열기, 스냅샷 저장, 시스템 명령 등 다양한 작업을 자연어에 가까운 명령으로 수행할 수 있도록 한다.

### 1.3 전체 구성

이후 구성은 다음과 같다. 2장에서는 관련 선행연구를 검토한다. 3장에서는 Amadda의 설계 원리와 시스템 구성을 설명한다. 4장에서는 실제 구현 내용과 결과를 분석한다. 5장에서는 결론과 함께 프로젝트를 통해 얻은 경험과 향후 개선 방향을 정리한다.

## 2. 관련 연구

컨텍스트 인식 컴퓨팅은 사용자의 현재 상황을 파악하여 적절한 정보와 서비스를 제공하는 연구 분야로, 모바일 환경과 지능형 사용자 지원 시스템의 발전과 함께 꾸준히 확장되어 왔다. 서효석과 이상용은 컨텍스트 인식 기반 개인화 서비스에서 사용자 행동 패턴을 추론하는 모델을 제안하며, 단순한 현재 상태 수집을 넘어 사용자의 목적과 행동 흐름을 파악하는 것이 중요하다고 보았다[1]. 이는 본 작품이 단순 프로세스 목록 저장이 아니라 "작업 재진입을 돕는 단서"를 수집하고자 한 방향과 연결된다. 또한 김성림과 권준희는 컨텍스트 인식 환경에서 사용자의 문맥에 따라 필요한 정보 접근 방식이 달라져야 함을 보였으며[2], 이는 작업 복귀 시 일괄적인 기록 제공이 아니라 사용자가 실제로 필요로 하는 정보 중심의 맥락 제공이 중요하다는 점을 시사한다.

국외 연구에서는 작업 전환과 재개를 지원하는 인터페이스 연구가 활발히 이루어졌다. Robertson 등은 `Task Gallery`를 통해 창과 문서를 작업 단위로 재배치하는 3차원 윈도우 관리 방식을 제안하였고, 사용자가 창 목록이 아닌 작업 단위로 환경을 인식한다는 점을 보여주었다[4]. Dragunov 등은 `TaskTracer`를 제안하여 다중 작업을 수행하는 지식 노동자의 데스크톱 활동을 추적하고 작업별로 자원을 분류하는 환경을 구현하였다[5]. Morris 등은 `SearchBar`를 통해 검색 기록, 방문 페이지, 메모를 통합 저장하여 task resumption과 정보 재탐색을 지원하였으며, 브라우저 탭의 URL과 제목을 구조적으로 저장하는 기능이 작업 재개에 실질적 도움이 될 수 있음을 보였다[6]. Bardram 등은 활동 중심 컴퓨팅(Activity-Based Computing)을 제안하면서, 전통적인 앱 중심 컴퓨팅은 사용자의 실제 활동 구조를 반영하지 못하며, 활동 단위를 기준으로 정보와 도구를 조직해야 한다고 주장하였다[7].

선행연구를 종합하면, 사용자의 작업 흐름 지원을 위해서는 단순 저장보다 문맥화된 기록이 중요하며[1][2][3], 작업 단위의 재구성과 재개를 지원하는 인터페이스가 필요하고[4][5], 브라우저 및 검색 활동 역시 중요한 작업 맥락 요소임을 알 수 있다[6]. 본 작품은 이러한 선행연구의 방향을 실제 Windows 데스크톱 환경에서 구현 가능한 로컬 도구로 구체화한 사례이다.

## 3. 제안 작품 소개

### 3.1 작품 개요

Amadda는 사용자가 작업 중이던 앱, 창, 브라우저 탭 정보를 하나의 스냅샷으로 저장하고, 이후 작업을 다시 시작할 때 그 정보를 활용해 빠르게 이전 흐름에 재진입하도록 돕는 데스크톱 기반 시스템이다. 완전한 세션 복제를 목표로 하지 않으며, 사용자가 "무엇을 하고 있었는지"를 즉시 떠올리게 만드는 작업 단서를 수집하고 구조화하는 데 집중한다.

핵심 설계 철학은 세 가지이다. 첫째, **로컬 중심 설계**를 통해 개인정보가 외부 서버로 전송되지 않도록 한다. 둘째, **자동 수집과 능동적 상호작용의 결합**으로, 사용자가 아무것도 하지 않아도 주기적으로 컨텍스트가 쌓이고, 필요할 때 음성이나 텍스트로 조회하거나 복구할 수 있다. 셋째, **공통 저장 구조**를 통해 프로세스, 창, 브라우저 탭, 음성 명령 등 다양한 입력 소스를 단일한 스키마 아래 관리하여 확장성을 확보한다.

### 3.2 이론적 배경

컨텍스트 인식 관점에서, `chrome.exe`가 실행 중이라는 사실만으로는 사용자의 작업 목적을 알기 어렵다. 그러나 특정 탭 제목, URL, 함께 열린 개발 도구의 창 제목을 결합하면 작업 맥락을 구체적으로 파악할 수 있다. 따라서 본 작품은 프로세스명, 창 제목, URL, 로컬 경로(`path`)와 같은 속성을 함께 저장하며, VS Code 창에서는 현재 열린 워크스페이스 경로를, Word 창에서는 문서명을 추출하여 보강한다[1][2].

활동 중심 컴퓨팅 관점에서, 사용자는 발표 자료 작성, 보고서 조사, 프로그래밍 디버깅처럼 작업 단위로 사고하지만 기존 환경은 앱과 파일 단위로 인터페이스를 제공한다[7]. Amadda는 여러 앱과 탭을 하나의 스냅샷 안에 담아 분산된 정보를 하나의 작업 흐름으로 묶어 재인식할 수 있도록 한다.

### 3.3 시스템 구성

Amadda는 다음 7개의 계층으로 구성된다.

**데스크톱 UI (PySide6):** 텍스트 명령 입력, 음성 입력 버튼, 로그 출력, Tracked Apps 선택, 스냅샷 히스토리 조회, 자동 스냅샷 토글을 제공하는 메인 인터페이스이다.

**명령 처리 계층:** 규칙 기반 intent parser와 dispatcher로 구성된다. parser는 자연어 입력을 `save_snapshot`, `restore_latest_snapshot`, `open_url`, `open_app`, `summarize`, `sleep`, `shutdown` 등의 intent로 변환한다. 한국어 표현과 영어 표현을 모두 지원하며, 사이트 이름 기반 URL 단축어, 앱 이름 기반 실행, 검색어 추출을 포함한다. dispatcher는 intent를 적절한 액션 서비스로 연결한다.

**스냅샷 수집 계층:** Windows 환경에서 실행 중인 프로세스와 가시 창 제목을 수집한다. `psutil`로 프로세스 목록을 열거하고, Win32 `EnumWindows` API로 가시 창 제목을 수집한다. 사용자가 Tracked Apps 인터페이스에서 선택한 프로세스만 최종 저장 대상으로 삼아 잡음을 줄인다. 또한 VS Code 창에서는 `%APPDATA%\Code\User\globalStorage\storage.json`을 참조해 워크스페이스 절대 경로를 보강하고, Word 창에서는 창 제목에서 문서명을 추출한다.

**음성 입력 계층:** `sounddevice`로 마이크 오디오를 스트리밍 수집하고, `silero-vad`로 음성 활동 감지(VAD)를 수행하여 말이 끝나면 자동으로 녹음을 종료한다. 수집된 오디오는 로컬 `openai-whisper` (small 모델)로 한국어 음성 인식을 수행한다. 모든 처리가 로컬에서 완결되어 외부 API 비용이 없다.

**로컬 API (FastAPI):** 데스크톱 앱과 함께 백그라운드 스레드로 실행된다. `/command`, `/snapshots/latest`, `/browser/snapshot` 엔드포인트를 제공하며, Chrome 확장이 탭 정보를 전송하는 진입점이기도 하다.

**데이터 저장 계층 (SQLite):** `snapshots` 테이블에 스냅샷 단위 메타데이터를, `snapshot_items` 테이블에 개별 항목을 저장한다. 각 항목은 `app_name`, `title`, `url`, `path`, `item_type`, `process_name`, `executable_path`, `created_at` 속성을 가진다. `url`은 브라우저 탭 URL에만, `path`는 VS Code 워크스페이스 경로나 Word 문서명 등 로컬 앱 컨텍스트에만 사용하여 의미를 명확히 구분한다. 스냅샷은 최대 288개(15분 간격 × 72시간)를 유지하고 오래된 것은 자동 삭제한다.

**브라우저 확장 계층 (Chrome Manifest V3):** `chrome.alarms` API로 15분 주기 자동 스냅샷을 전송하고, 아이콘 클릭으로 즉시 전송도 지원한다. `chrome.windows.getLastFocused`로 마지막 활성 창의 탭 목록을 수집하여 로컬 API로 전송한다.

### 3.4 데이터 흐름

**자동 스냅샷 저장:** 앱 시작 시 15분 타이머가 자동으로 활성화되고, 타이머가 발동할 때마다 Windows 프로세스 수집 결과와 최신 브라우저 탭 정보를 병합하여 하나의 스냅샷으로 저장한다. 브라우저 측에서는 Chrome 확장의 alarms이 15분마다 독립적으로 로컬 API에 탭 정보를 전송한다.

**작업 요약:** 사용자가 "뭐 하고 있었어?"라고 입력하거나 음성으로 말하면, 최근 저장된 스냅샷 3개를 불러와 창 제목, 경로, 브라우저 탭 정보를 하나의 프롬프트로 구성한 뒤 Gemini API에 전송한다. Gemini는 항목별로 사용자가 무엇을 하고 있었는지를 한국어로 구체적으로 서술한다.

**작업 복구:** 사용자가 복구 명령을 입력하면 최신 스냅샷의 브라우저 탭 URL을 기본 브라우저로 재오픈하고, VS Code 창에 저장된 워크스페이스 절대 경로가 있으면 `code <path>` 명령으로 VS Code를 해당 프로젝트와 함께 재실행한다.

### 3.5 설계상의 특징

**이기종 컨텍스트의 통합 저장:** 프로세스, 창, 브라우저 탭은 성격이 다르지만 `snapshot_items` 공통 테이블에서 관리된다. 새로운 컨텍스트 소스가 추가되어도 전체 구조를 바꾸지 않아도 된다.

**url과 path의 의미 구분:** 초기에는 URL과 로컬 경로를 같은 필드에 혼용했으나, 브라우저 URL과 로컬 앱 경로는 의미가 다르므로 별도 필드(`url`, `path`)로 구분하였다. 이를 통해 LLM 요약 프롬프트와 복구 로직이 더 정확하게 동작한다.

**사용자 선택 기반 tracked app:** 모든 프로세스를 저장하지 않고 사용자가 중요하다고 판단한 앱만 추적한다. 이는 저장 결과를 읽기 쉽게 만들고 LLM 요약 품질을 높이는 데 기여한다.

**로컬 전용 음성 처리:** 음성 인식(Whisper)과 VAD(silero-vad)를 모두 로컬에서 실행하여 외부 API 비용 없이 오프라인에서도 동작한다. VAD를 통해 고정 5초 대기 없이 말이 끝나는 시점을 자동 감지하므로 응답 지연이 크게 줄어든다.

## 4. 구현 및 결과 분석

### 4.1 구현 환경 및 주요 기술

본 작품은 Windows 11 환경에서 Python 3.11 기반으로 구현하였다. 주요 기술 스택은 다음과 같다.

| 구성 요소 | 기술 |
|---|---|
| 데스크톱 UI | PySide6 6.7 |
| 로컬 API 서버 | FastAPI + uvicorn |
| 데이터 저장 | SQLite3 (표준 라이브러리) |
| 프로세스 수집 | psutil + Win32 ctypes (EnumWindows) |
| 음성 인식 | openai-whisper (small, 로컬) |
| 음성 활동 감지 | silero-vad 5.x |
| 마이크 입력 | sounddevice |
| LLM 요약 | Google Gemini 2.5 Flash Lite (google-genai SDK) |
| 브라우저 확장 | Chrome Manifest V3, chrome.alarms |

### 4.2 Windows 스냅샷 수집

Windows 스냅샷 수집기는 프로세스 열거, 가시 창 수집, tracked app 필터링, 컨텍스트 보강의 4단계로 동작한다.

**프로세스 열거:** `psutil.process_iter()`로 실행 중인 프로세스의 PID, 이름, 실행 파일 경로를 수집한다. psutil이 없는 경우 `tasklist /fo csv` 명령으로 폴백한다.

**가시 창 수집:** Win32 `EnumWindows` API로 현재 보이는 모든 창의 핸들을 열거하고, `GetWindowTextW`로 창 제목을, `GetWindowThreadProcessId`로 해당 프로세스 PID를 수집하여 PID별 창 제목 목록을 구성한다.

**tracked app 필터링:** 수집된 프로세스 중 사용자가 Tracked Apps 인터페이스에서 선택한 프로세스만 최종 저장 대상으로 선정한다. Chrome이나 VS Code처럼 여러 서브프로세스를 생성하는 앱은 `(process_name, executable_path)` 기준으로 그룹화하여 중복 저장을 방지한다.

**컨텍스트 보강:** VS Code 창(Code.exe)의 경우, `%APPDATA%\Code\User\globalStorage\storage.json`을 읽어 폴더명과 절대 경로 매핑을 구성한다. 창 제목에서 "파일명 - 프로젝트명 - Visual Studio Code" 형식을 파싱하여 프로젝트명과 매핑된 경로를 `path` 필드에 저장한다. Word 창(WINWORD.EXE)의 경우, 창 제목에서 문서명을 추출하여 `path` 필드에 저장한다.

### 4.3 브라우저 스냅샷 자동 수집

Chrome 확장은 Manifest V3 기반 서비스 워커로 구현하였다. `chrome.alarms` API를 사용하여 15분 주기로 자동 스냅샷을 전송하며, 앱 설치 및 브라우저 시작 시 알람이 자동 등록된다. 탭 수집 시에는 `chrome.windows.getLastFocused({ populate: true, windowTypes: ["normal"] })`를 사용하여 마지막으로 활성화된 일반 창의 탭 목록을 수집한다. 서비스 워커는 UI가 없어 "현재 창"의 개념이 없으므로, `currentWindow: true` 방식 대신 이 방법을 사용한다.

수집된 탭 정보는 `POST /browser/snapshot` 엔드포인트로 전송되며, 백엔드는 각 탭을 `item_type="browser_tab"`으로 저장한다. 데스크톱 GUI에서 스냅샷 저장 시에는 최신 브라우저 탭 스냅샷을 함께 병합하여 단일 스냅샷에 Windows 프로세스 정보와 브라우저 탭 정보가 함께 포함되도록 한다.

### 4.4 자동 주기 스냅샷

데스크톱 앱에서는 PySide6의 `QTimer`를 사용하여 15분 주기로 자동 스냅샷을 저장한다. 앱 시작 시 타이머가 자동으로 활성화되어 별도의 설정 없이도 즉시 작동한다. UI의 `Auto: ON (15m)` 버튼으로 언제든 활성화·비활성화할 수 있으며, 자동 저장 시 로그 패널에 저장 결과가 표시된다.

브라우저 확장의 alarms와 데스크톱 앱의 QTimer가 독립적으로 동작하므로, 브라우저 탭 정보는 확장이 주기적으로 로컬 API에 기록하고, 데스크톱 앱은 그 최신 기록을 참조하여 스냅샷 저장 시 병합한다.

### 4.5 LLM 기반 작업 요약

사용자가 "뭐 하고 있었어?", "요약해줘" 등을 입력하면 최근 3개의 스냅샷에서 창 제목(`path` 포함)과 브라우저 탭(`title`, `url`)을 추출하여 구조화된 텍스트로 구성하고, Gemini 2.5 Flash Lite 모델에 한국어 요약을 요청한다. 프롬프트는 파일명, 강의명, 영상 제목 등 구체적인 항목을 빠짐없이 언급하도록 규칙을 명시하여 요약 품질을 높인다.

Gemini 2.5 Flash Lite는 Google AI Studio에서 무료로 발급되는 API 키로 사용할 수 있어 추가 비용이 없다. API 키는 환경변수로 관리하며 코드에 하드코딩하지 않는다. 단일 스냅샷보다 3개 스냅샷 집계를 사용했을 때 맥락이 풍부해져 요약 품질이 크게 향상됨을 확인하였다.

### 4.6 음성 입력 (Whisper + VAD)

음성 입력은 마이크 녹음, VAD 기반 종료 감지, Whisper 음성 인식의 3단계로 동작한다.

**녹음 및 VAD:** `sounddevice.InputStream`으로 16kHz 모노 오디오를 512 샘플(약 32ms) 단위로 실시간 수집한다. 각 청크를 `silero-vad` VADIterator에 전달하여 음성 시작(`start`)과 종료(`end`) 이벤트를 감지한다. 음성 종료 이벤트는 0.8초 침묵이 지속된 후 발생하며, 최소 0.5초 이상의 발화가 있어야 유효한 음성으로 처리하여 소음에 의한 오감지를 방지한다. 최대 10초의 안전 상한도 적용한다.

**음성 인식:** 수집된 오디오 전체를 `openai-whisper` small 모델에 전달하여 한국어 음성을 텍스트로 변환한다. `fp16=False` 옵션으로 GPU 없이도 CPU에서 안정적으로 동작한다. 변환된 텍스트는 텍스트 입력과 동일하게 parser를 거쳐 intent로 처리된다.

VAD 적용으로 기존 고정 5초 대기가 제거되어, 짧은 명령어는 말이 끝난 후 약 1~2초 내에 처리가 시작된다. 음성 처리 전 과정이 로컬에서 완결되므로 외부 API 비용이 없다.

### 4.7 명령어 시스템

명령어 파서는 규칙 기반으로 구현되어 있으며, 텍스트 입력과 음성 인식 결과 모두에 동일하게 적용된다. 인식 가능한 명령 범주는 다음과 같다.

**스냅샷 관련:** "저장해줘", "save snapshot" → 현재 컨텍스트 스냅샷 저장. "어제 하던 거 다시 열어줘", "restore latest snapshot" → 최신 스냅샷 복구.

**작업 요약:** "뭐 하고 있었어", "요약해줘", "summarize" → Gemini 기반 작업 요약.

**사이트 열기:** "유튜브 켜줘", "구글 열어줘", "네이버 켜줘" 등 → 해당 URL을 기본 브라우저로 오픈.

**앱 실행:** "워드 켜줘", "커맨드 창 켜줘", "vscode 켜줘" 등 → 해당 앱 실행.

**웹 검색:** "파이썬 검색해줘", "유튜브에서 로파이 검색해줘", "네이버에서 날씨 찾아줘" → 검색어를 추출하여 해당 검색 엔진에서 결과 오픈.

**시스템:** "절전해줘", "컴퓨터 꺼줘" → 사용자 확인 후 실행.

위험 명령(절전, 종료)은 `PermissionService`가 감지하여 GUI에서 사용자 확인 대화상자를 표시한 후에만 실행된다.

### 4.8 스냅샷 히스토리 UI 및 복구 개선

**히스토리 UI:** `History` 버튼을 누르면 과거 스냅샷 목록 대화상자가 열린다. 각 스냅샷의 저장 시각, 창 항목 수, 브라우저 탭 수를 한눈에 확인할 수 있으며, 행을 선택하고 `Restore Selected`를 누르면 해당 시점의 컨텍스트를 복구한다.

**복구 개선:** 기존에는 브라우저 URL 재오픈만 지원하였으나, VS Code 창 항목 중 절대 경로가 저장된 경우 `code <path>` 명령으로 해당 워크스페이스를 함께 재오픈하도록 개선하였다. 이를 통해 브라우저 탭과 VS Code 프로젝트를 한 번에 복구할 수 있다.

### 4.9 결과 및 분석

구현 결과, Amadda는 자동 수집부터 요약, 복구, 명령 실행까지 하나의 도구로 통합된 작업 컨텍스트 복구 시스템을 완성하였다. 주요 성과를 분석하면 다음과 같다.

**자동 수집의 실용성:** 앱 시작과 동시에 15분 주기 스냅샷이 활성화되어 사용자가 별도로 저장 명령을 내리지 않아도 컨텍스트가 축적된다. 브라우저 탭 정보도 별도 조작 없이 15분마다 자동 전송된다.

**LLM 요약의 유용성:** 최근 3개 스냅샷을 집계하여 Gemini에 요약을 요청하면, 구체적인 파일명, 탭 제목, VS Code 프로젝트명이 포함된 한국어 요약이 생성된다. 단일 스냅샷 대비 맥락이 풍부해져 요약 품질이 크게 향상됨을 확인하였다.

**음성 입력의 반응성:** VAD 적용 전에는 말의 길이와 무관하게 5초를 기다려야 했으나, VAD 적용 후에는 말이 끝난 뒤 0.8초 침묵을 감지하여 자동으로 녹음이 종료된다. 짧은 명령어의 경우 총 처리 시간이 기존 5초+에서 약 2초+로 단축된다.

**컨텍스트 보강의 의미:** `path` 필드를 별도로 분리하고 VS Code 워크스페이스 경로를 저장함으로써, LLM 요약 프롬프트에서 단순 창 제목보다 구체적인 작업 경로가 제공된다. 예를 들어 "Visual Studio Code"라는 창 제목만 있을 때보다 "C:\graduation\Amadda (경로)"가 함께 있을 때 요약의 구체성이 높아진다.

**한계:** Word 문서 복구는 창 제목에서 추출한 문서명만 저장되므로 전체 경로를 알 수 없어 파일을 자동으로 열지 못한다. 또한 앱 내부 편집 상태(스크롤 위치, 커서 위치 등)는 복원되지 않는다. 브라우저는 Chrome만 지원하며, Firefox나 Edge는 별도 확장이 필요하다.

## 5. 결론 및 소감

본 작품 `Amadda`는 사용자가 중단한 작업에 빠르게 재진입할 수 있도록 돕는 로컬 기반 컨텍스트 복구 시스템이다. Windows 프로세스 및 창 제목 자동 수집, Chrome 브라우저 탭 주기 수집, VS Code·Word 컨텍스트 보강, LLM 기반 작업 요약, 오프라인 음성 입력, 규칙 기반 명령어 시스템, 스냅샷 히스토리 UI, 브라우저 탭과 VS Code 워크스페이스 복구를 하나의 도구로 통합하였다.

본 작품의 의의는 "완전한 세션 복제"가 아닌 "실용적인 작업 단서 수집과 복구"라는 목표에 집중한 점이다. 사용자가 이전 작업을 즉시 떠올릴 수 있도록 핵심 단서를 자동으로 축적하고, 필요할 때 음성이나 텍스트로 바로 조회하거나 복구할 수 있도록 설계하였다. 또한 모든 기능이 로컬에서 동작하여 개인정보 유출 위험 없이 사용할 수 있다는 점도 중요한 특징이다.

프로젝트를 진행하면서 가장 크게 느낀 점은, 사용자에게 실제로 의미 있는 정보와 기술적으로 수집하기 쉬운 정보가 항상 일치하지 않는다는 것이다. 프로세스 목록 수집은 단순했지만, 사용자가 실제로 필요한 것은 "어떤 프로젝트를 작업하고 있었는지", "어떤 탭에서 무엇을 보고 있었는지"라는 더 구체적인 문맥이었다. 이 간극을 좁히기 위해 VS Code 워크스페이스 경로 추출, `url`과 `path`의 구분, LLM 요약 프롬프트 설계 등의 추가 작업이 필요했으며, 이 과정이 단순 구현보다 더 많은 고민을 요구했다.

또한 데스크톱 UI, 로컬 API, 브라우저 확장, SQLite, LLM, 음성 인식을 하나의 작품으로 통합하는 과정에서 각 구성 요소의 책임 분리와 인터페이스 설계의 중요성을 체감하였다. 초기에는 기능 추가에 집중하기 쉬웠지만, 프로젝트가 진행될수록 저장 구조를 단순하게 유지하고 각 계층의 역할을 명확히 구분하는 것이 장기적으로 유지 보수와 확장을 쉽게 만든다는 점을 경험으로 배웠다.

향후 개선 방향으로는 Word 문서 전체 경로 추적, 멀티 브라우저 지원, 더 세밀한 복구 UI(스냅샷 항목별 선택 복구), 스냅샷 간 작업 흐름 자동 분석 등을 생각해 볼 수 있다.

종합하면, Amadda는 "아, 맞다, 거기서 뭐 하고 있었지"라는 순간을 대신 기억해 주는 도구라는 출발점에서 시작하여, 실제로 동작하는 통합 컨텍스트 복구 시스템으로 완성되었다. 단순한 아이디어 제시를 넘어 실제 구현, 구조 설계, 기능 한계 분석까지 수행한 경험은 이후 소프트웨어 개발에서 큰 자산이 될 것이다.
